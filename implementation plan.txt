Plan de Batalla: Ware-Intelligence Engine v2.0
Misión: Evolucionar el Ware-Intelligence Engine de una herramienta de análisis de archivos a una plataforma de inteligencia operativa con un sistema dual: conectores directos a bases de datos y la capacidad de carga manual de archivos como opción secundaria.
Fase 1: Cimientos y Arquitectura del Backend (El Chasis)
Objetivo: Construir el núcleo modular y reutilizable para la extracción de datos sin tocar la interfaz de usuario. Esta es la base sobre la que se construirá todo lo demás.
1. Crear el Módulo de Conectores (src/connectors.py):
   * Define una clase base abstracta DataConnector. Esta clase forzará una estructura estándar para todos los conectores futuros.
from abc import ABC, abstractmethod
import pandas as pd

class DataConnector(ABC):
   @abstractmethod
   def connect(self):
       pass

   @abstractmethod
   def get_data(self) -> pd.DataFrame:
       # Este método SIEMPRE debe devolver un DataFrame de Pandas.
       pass

   @abstractmethod
   def disconnect(self):
       pass

   * Crea tu primer conector concreto: SQLConnector(DataConnector).
   * Su constructor (__init__) recibirá las credenciales de la base de datos (host, usuario, pass, db) y el mapeo de columnas.
   * El método get_data usará SQLAlchemy y pd.read_sql() para ejecutar una consulta y devolver el DataFrame estructurado que tu motor run_engine ya entiende.
   * Actualiza requirements.txt: Añade SQLAlchemy y el driver para tu base de datos de prueba (ej. mysql-connector-python).
   2. Pruebas Unitarias Aisladas:
   * Crea un script de prueba (tests/test_connectors.py) que no dependa de Flask.
   * Este script debe importar SQLConnector, instanciarlo con credenciales de una base de datos local (Docker es tu amigo aquí) y afirmar que el DataFrame devuelto es correcto en forma y contenido.
   * No avances hasta que esta prueba pase. Un cimiento débil derrumba el edificio.
Fase 2: Rediseño del Flujo de Usuario y la Interfaz (La Carrocería)
Objetivo: Modificar la experiencia del usuario para que la elección del método de análisis sea clara e intencional.
   1. Rediseñar la Página de "Nuevo Análisis" (index.html):
   * Esta página ya no es un formulario. Es un centro de decisión.
   * Presenta dos opciones visualmente claras:
   * Opción A: "Analizar desde Archivo": Un botón que, al hacer clic, revela el formulario de carga de archivos que ya tienes (puedes usar JavaScript para mostrarlo/ocultarlo).
   * Opción B: "Analizar desde Conexión Directa": Un botón que redirige al usuario a una nueva página para gestionar sus conexiones (/connections).
   2. Crear la Interfaz de Gestión de Conexiones (templates/connections.html):
   * Esta página mostrará una lista de las conexiones que el usuario ya ha configurado.
   * Tendrá un botón prominente: "Crear Nueva Conexión".
   * Al hacer clic, se mostrará un formulario para:
   * Nombre de la Conexión (ej. "WMS Bodega Principal").
   * Tipo de Conector (un desplegable, por ahora solo con "MySQL").
   * Campos para las credenciales (Host, Usuario, Contraseña, etc.).
   * Campos para el mapeo de datos (Nombre de la tabla, y mapeo de columnas de esa tabla a tus campos estándar).
Fase 3: Integración, Seguridad y Lógica de la Aplicación (El Motor y el Cableado)
Objetivo: Conectar el nuevo backend con el nuevo frontend, manejar las credenciales de forma segura y unificar los flujos de procesamiento.
   1. Ampliar la Base de Datos (app.py):
   * Define un nuevo modelo en SQLAlchemy: DataSource.
   * Campos: id, user_id, name, type ('mysql', 'api'), y un campo para las credentials.
   * Seguridad No Negociable: Las credenciales DEBEN estar cifradas en la base de datos.
   * Añade cryptography a tu requirements.txt.
   * Crea funciones de utilidad (encrypt_data, decrypt_data) que usarás antes de guardar en la BBDD y después de leer de ella. No guardar credenciales en texto plano es una regla de hierro.
   2. Implementar las Nuevas Rutas en Flask (app.py):
   * GET /connections: Obtiene y muestra las conexiones del usuario actual.
   * POST /connections/create: Recibe los datos del formulario, cifra las credenciales y guarda la nueva DataSource en la BBDD.
   * POST /connections/<id>/delete: Elimina una conexión.
   3. Crear la Nueva Ruta de Procesamiento para Conectores:
   * Crea una ruta POST /process_connection.
   * Esta ruta recibirá el ID de la DataSource a utilizar.
   * Su lógica será:
   1. Buscar la DataSource en la BBDD por su ID.
   2. Verificar que pertenece al usuario actual.
   3. Descifrar las credenciales.
   4. Instanciar el SQLConnector con esas credenciales y el mapeo.
   5. Llamar al método get_data() para obtener el DataFrame.
   6. Pasar el DataFrame a tu función run_engine.
   7. Guardar los resultados en AnalysisReport y redirigir a la página de resultados.
   4. Asegurar la Convergencia:
   * Verifica que tanto el flujo de carga de archivos como el nuevo flujo de conectores terminan en el mismo lugar: guardan un reporte y redirigen a /report/<report_id>. La experiencia del usuario post-análisis debe ser idéntica.
Fase 4: Expansión Futura (El Siguiente Nivel)
Objetivo: Asegurar que la arquitectura esté lista para crecer.
   1. Implementar Conector de API REST:
   * Una vez que el sistema dual funcione, el siguiente paso es crear un APIConnector(DataConnector).
   * Usará la librería requests para obtener datos JSON y pandas.json_normalize para convertirlos en un DataFrame.
   * El patrón ya está establecido. Solo tienes que añadir "API REST" al desplegable de tipo de conector y ajustar el formulario y la lógica para manejar una URL y una API Key (que también debe ser cifrada).
